# 即時字幕功能開發與優化完整交接文件

**日期**: 2026-02-09
**版本**: v2.1
**目標**: 提供「即時字幕功能核心開發」、「懸浮視窗 UI 優化」及「穩定性與準確度提升」的完整技術細節。

---

## 1. 專案概述

本文件整合了 VoiceInk 專案中即時字幕功能的所有開發歷程與技術決策。我們完成了一個基於 Electron 的系統音訊擷取、轉錄與即時顯示功能，並針對 Windows 平台進行了深度優化。

---

## 2. 核心問題與解決策略 (Core Implementation)

### 2.1 音訊處理與 API 整合

針對 OpenRouter API 與瀏覽器錄音的限制，我們實作了以下解決方案：

| 問題範疇                | 問題描述                                  | 解決方案                                                                                                              |
| ----------------------- | ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **音訊格式**      | OpenRouter API 不支援 WebM 格式。         | **前端轉碼**：實作 `webmToWav`，在送出請求前將瀏覽器的 WebM 錄音轉為標準 WAV (16-bit PCM)。                   |
| **API 限制**      | 請求頻率過高導致 429 錯誤。               | **請求節流**：將錄製區塊時長 (`CHUNK_DURATION`) 設為 3000ms，並加入 `isProcessing` 鎖定機制，防止請求堆疊。 |
| **錄製中斷**      | 處理錯誤導致錄製循環停止。                | **非阻塞循環**：將 `processAudioChunkData` 與錄製循環解耦，並完善錯誤捕獲 (`.catch`)。                      |
| **MediaRecorder** | `timeslice` 模式產生的 Blob 缺 Header。 | **完整錄製**：放棄 `timeslice`，改為每次錄製都建立新的 `MediaRecorder` 實例，確保獲得完整檔案標頭。         |

### 2.2 穩定性與準確度優化 (Optimization & Hardening)

針對靜音時 AI 容易「無中生有」（Hallucination）的問題，我們建立了「多層防護網」：

| 層級                        | 防護機制               | 參數/邏輯                                                                                                                                                                                            |
| --------------------------- | ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **第 1 層 (VAD)**     | **強化靜音檢測** | **RMS 閾值**: `0.015` -> **`0.025`** (過濾底噪)`<br>`**語音佔比**: `0.05` -> **`0.12`** (需 12% 以上語音活動才發送)                                                |
| **第 2 層 (Context)** | **上下文累積**   | 將 `previousTranscript` (單句) 改為 **`transcriptHistory` (陣列)**，保留最近 10 句，避免斷章取義。                                                                                         |
| **第 3 層 (Prompt)**  | **提示詞工程**   | **人聲優先**：明確指示 AI 只有聽到人聲才轉錄。`<br>`**合法出口**：允許輸出 `（音樂）`、`（掌聲）` 標籤以宣洩判定結果。`<br>`**防呆**：強調「不確定就留白」。               |
| **第 4 層 (Filter)**  | **後處理過濾**   | **長度檢測**：3秒片段 > 60字視為編造，丟棄。`<br>`**標籤過濾**：前端直接過濾掉 `（掌聲）` 等獨立標籤不顯示。`<br>`**模式過濾**：過濾 `謝謝觀看`、`歡迎訂閱` 等常見幻覺。 |

## 3. 懸浮視窗 UI 優化 (Floating Window UI)

針對 Windows 平台上的視窗渲染異常與互動問題進行了解決：

| 問題                        | 原因                                                                                    | 解決方案                                                                                                                                   |
| --------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **白色標題列殘留**    | Electron 在 Windows 上同時使用 `transparent: true` 與 `frame: false` 時的渲染 Bug。 | **最終方案**：改用不透明視窗 (`transparent: false`)，背景設為黑色 (`#1a1a1a`) 徹底解決渲染異常。                                 |
| **無法拖動視窗**      | 內容區域佔滿全視窗且設定了 `no-drag`。                                                | **分離拖動區**：在頂部新增專用的 `.drag-area` (`-webkit-app-region: drag`)，內容區域恢復正常互動。                               |
| **無法調整大小/滾動** | 透明視窗在 Windows 上會強制禁用 `resizable` 且影響滑鼠事件。                          | **放棄透明度**：改用不透明背景，恢復 `resizable: true` 與正常的滑鼠滾動功能。                                                      |
| **關閉狀態不同步**    | 懸浮視窗關閉後主介面未感知。                                                            | **IPC 雙向同步**：`<br>`1. Main: 監聽視窗關閉 -> 發送 `subtitle:closed`。`<br>`2. Renderer: 監聽事件 -> 觸發 `stopCapture`。 |

**視窗配置 (`src/main/main.js`)**:

```javascript
subtitleWindow = new BrowserWindow({
  frame: false,
  transparent: false, // 關鍵：使用不透明解決 Windows 相容性
  backgroundColor: '#1a1a1a',
  resizable: true,    // 關鍵：恢復調整大小
  autoHideMenuBar: true, // 避免選單列出現
  // ...
})
subtitleWindow.setMenu(null) // 雙重保險
```

## 4. 資料流與處理邏輯

```
[錄音 3秒]
    |
    v
<靜音檢測 RMS:0.025 Ratio:0.12>
    |
    +--[低於閾值]--> [丟棄 (不發 API)]
    |
    +--[通過]--> [發送 API (帶 10 句歷史上下文)]
                     |
                     v
               [AI 轉錄回應]
                     |
                     v
            <長度檢查 超過 60 字?>
                     |
                     +--[是]--> [丟棄 (視為編造)]
                     |
                     +--[否]--> <內容檢查>
                                  |
                                  +--[純標籤 (音樂/掌聲)]--> [丟棄 (過濾不顯示)]
                                  |
                                  +--[常見幻覺 (謝謝觀看)]--> [丟棄]
                                  |
                                  +--[正常人聲]--> [顯示並加入歷史陣列]
```

## 5. 開發建議與注意事項

1. **Windows 透明視窗限制**：

   - 除非必要，盡量避免在 Windows 使用 `transparent: true`，因為它會導致 `resizable` 失效、滑鼠穿透問題及渲染 Bug。
   - 若必須使用，需處理複雜的 event forwarding。
2. **瀏覽器錄音坑**：

   - 不同瀏覽器的 `MediaRecorder` 實作差異大，始終假設 `timeslice` 產生的 Blob 不完整。
   - 建議前端自行轉碼為 WAV (PCM) 以確保相容性。
3. **VAD (語音活動檢測)**：

   - **不要依賴 AI** 判斷靜音，這非常昂貴且容易產生幻覺。
   - 務必在客戶端進行訊號層級的過濾（RMS/Zero-crossing rate）。
   - 目前閾值 (`0.025` / `0.12`) 是針對一般說話音量調整的，若使用者反應小聲說話被吃掉，可微調降至 `0.02` / `0.10`。
4. **API 延遲觀察**：

   - 目前每 3 秒發送一次請求，加上上下文長度增加，API 回應可能會稍慢。若延遲過高，可考慮減少上下文句數 (`MAX_HISTORY_COUNT` 從 10 降至 5)。
